# 02 用户分层与核心场景地图

## 1. 目标用户分层

为了避免功能泛化，先定义主要服务对象（按优先级）：

### 1.1 A 类用户：独立开发者 / 小团队技术负责人

特征：
- 既要写代码又要做产品决策。
- 时间碎片化，频繁在“想法-实现-上线”切换。
- 容易在技术细节中偏离产品目标。

核心需求：
- 快速做可用版本。
- 降低决策焦虑（选技术栈、选方案）。
- 留下可复盘的成长记录。

### 1.2 B 类用户：AI 辅助开发新手

特征：
- 会写部分代码，但缺完整项目经验。
- 常见问题是“能跑起来，但越改越乱”。

核心需求：
- 结构化指导而不是只给代码。
- 看得懂“为什么这样做”。
- 避免重复踩坑。

### 1.3 C 类用户：内容创作者/教学者

特征：
- 需要可演示、可讲述、可分享的项目过程。

核心需求：
- 有故事线的项目轨迹。
- 可导出的决策和复盘材料。

## 2. 关键使用场景（Jobs To Be Done）

### 场景 1：从 0 到 1 启动项目

用户任务：
- 把一个模糊想法变成可运行雏形。

典型痛点：
- 目标不清，越做越偏。
- 技术选择过多，难以取舍。

机会点：
- 初心锚点 + 分岔对比，先收敛目标再落地代码。

### 场景 2：中途卡住与返工

用户任务：
- 遇到报错、需求变化、架构混乱时继续推进。

典型痛点：
- 修 bug 依赖运气，不形成方法。
- 看不到过去决策为何做错。

机会点：
- 失败博物馆沉淀“问题模式 -> 修复策略”。

### 场景 3：准备上线或对外展示

用户任务：
- 向团队、客户、用户解释项目价值与决策过程。

典型痛点：
- 只能展示结果，无法解释过程。
- 缺少可信的迭代轨迹。

机会点：
- 人生线输出“目标-方案-取舍-结果”的完整链路。

## 3. 用户价值地图（功能与价值对照）

### 3.1 功能到价值映射

- 初心锚点：减少偏航，强化目标一致性。
- 分岔人生：降低决策成本，减少后悔性重构。
- 失败博物馆：把一次错误转化为长期资产。
- 人生线：提升可展示性与长期成就感。

### 3.2 情绪价值映射

- 可控感：我知道自己在做什么。
- 成长感：我每周都比上周更稳。
- 归属感：这不是工具，是我的创作搭档。

## 4. 用户旅程（简版）

1. 进入会话，写下初心（目标/受众/不做什么）。
2. 系统给出 2-3 套可行路径，显示成本与风险。
3. 用户选择路径并执行开发。
4. 遇到错误时系统自动归档失败类型与修复方案。
5. 会话结束后生成成长摘要与下一步建议。

## 5. 对应的优先级结论

如果只做 1 个季度，优先顺序建议：
1. 初心锚点（价值密度高、实现成本低）。
2. 分岔人生（直接体现“不只是写代码”）。
3. 失败博物馆（增强留存与复购潜力）。
4. 人生线（作为品牌化展示层）。
