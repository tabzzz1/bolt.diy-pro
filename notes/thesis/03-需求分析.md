# 第三章 需求分析

## 3.1 系统建设目标

### 3.1.1 平台功能目标

本文所研究的平台面向“自然语言驱动的全栈应用辅助生成与验证”场景，其核心目标并不是单纯提供一个聊天界面，也不是只实现代码片段补全，而是构建一个能够承接完整开发闭环的智能辅助系统。具体而言，系统需要支持用户以自然语言描述需求，由平台完成需求理解、模型调用、代码生成、文件写入、依赖安装、运行预览和结果反馈等一系列操作，并允许用户在此基础上继续通过对话或手工编辑进行迭代。

基于这一总体目标，平台应满足以下功能导向要求。第一，系统应具备自然语言交互能力，能够稳定接收用户输入并将其转换为结构化生成任务。第二，系统应具备项目级代码生成能力，而不仅限于零散代码片段输出。第三，系统应具备浏览器内运行与预览能力，使生成结果可以在当前工作台中直接验证。第四，系统应具备对话、编辑、预览联动能力，支持用户在一个统一界面中完成需求表达、代码观察和运行反馈分析。第五，系统应具备历史记录保存、模型管理、第三方服务接入和部署扩展能力，以提升系统的完整性和持续使用价值。

### 3.1.2 平台性能目标

由于本系统服务的典型场景是原型构建、学习实践和轻量级开发辅助，因此性能目标应围绕“交互连续性”和“反馈及时性”展开，而不只是单纯追求吞吐量。首先，系统在用户提交自然语言需求后，应能够较快给出流式响应或阶段性状态反馈，避免长时间无反馈引发的等待焦虑。其次，在代码生成完成后，平台应能较快完成文件写入、依赖安装和开发服务器启动，并将结果呈现在预览区域。再次，在文件内容切换、编辑器操作、对话滚动和视图切换过程中，系统应保持界面交互流畅，避免明显卡顿。

结合项目实现特点，平台还应保证工作台内部状态更新的稳定性。例如，在模型切换、聊天历史恢复、项目快照回滚和差异视图切换等场景下，系统应能正确维护界面状态一致性；在 WebContainer 启动、终端输出刷新和预览端口切换时，系统应尽量降低无效等待与重复执行。

### 3.1.3 平台应用目标

本系统的应用目标主要体现在三个方面。第一，面向初学者和教学场景，系统希望降低前端与全栈学习中的环境门槛，使用户可以更专注于需求表达、界面理解和结果观察。第二，面向产品构思与原型验证场景，系统希望缩短从想法到可运行演示页面之间的时间路径，提升创意转化效率。第三，面向智能开发工具研究与实现场景，系统希望验证“自然语言驱动 + 浏览器端运行时 + 可视化工作台”这一技术路线的工程可行性，为相关平台提供参考。

## 3.2 系统功能需求分析

### 3.2.1 自然语言输入与对话交互需求

自然语言交互是本系统的入口能力。平台需要支持用户以文本形式描述需求，并能够持续接收多轮补充说明、修正意见和局部优化指令。由于开发需求往往并非一次性完整给出，系统应支持多轮会话上下文保持，使后续生成能够参考前文语义和当前项目状态。

从项目现有实现看，聊天模块不仅承担消息输入与展示功能，还承担模型选择、提示增强、消息解析、工具调用结果显示和错误反馈提示等职责。因此，系统在对话交互层面至少应满足以下要求：能够稳定显示用户消息与助手消息；能够按流式方式展示模型响应；能够在消息中嵌入结构化产物卡片与工具调用卡片；能够对异常调用给出明确提示；能够在需要时插入模板提示、URL 内容和语音识别文本等附加输入。

### 3.2.2 项目代码生成需求

与传统聊天机器人不同，本系统的输出目标不是一般性文本问答，而是可运行的项目级代码产物。因此，系统必须支持根据用户描述生成文件集合、目录结构、依赖声明、构建命令和启动命令，并保证这些内容能够被运行时模块接收和执行。

这一需求可进一步拆解为三个层面。其一，系统应能根据用户需求选择合适的生成策略和项目模板，避免每次都从完全空白状态开始。其二，系统应能基于已有文件上下文增量生成内容，而不是在多轮对话后反复输出互相冲突的完整文件。其三，系统应支持在生成过程中将文件操作、命令操作和构建操作以可追踪形式传递给工作台，使用户知道系统做了什么、改了什么、当前运行到哪一步。

### 3.2.3 浏览器内编译运行需求

要使平台真正形成闭环，仅有代码生成是不够的。系统还必须具备在浏览器内写入项目文件、安装依赖、执行命令并启动开发服务的能力。这一能力是平台区别于一般 AI 对话工具的重要特征。

对应到需求层面，系统应支持以下功能：一是能够在浏览器端构建项目文件系统；二是能够执行依赖安装和脚本命令；三是能够监控构建与运行结果，并将输出同步至终端或状态提示区域；四是能够把运行后的应用通过预览面板展示给用户；五是在运行失败时，能够将错误信息反馈给对话区或工作台，以支撑后续修复。

### 3.2.4 实时预览与热更新需求

系统面向的是交互式开发体验，因此用户不仅需要看到生成结果，还需要快速验证结果是否符合预期。为此，平台应提供实时预览能力，并在代码变化后尽可能快地刷新显示内容。

该需求包含三个关键点。首先，预览区域应能稳定加载当前运行中的应用，并支持端口选择和地址切换。其次，当文件变更或重启服务后，预览结果应及时更新，使用户能够感知修改效果。再次，系统应在预览发生异常时提供清晰提示，例如运行时报错、资源加载失败或预览消息异常等，以便用户和模型继续迭代修正。

### 3.2.5 代码编辑与文件管理需求

虽然平台以自然语言驱动为主，但实际开发过程中用户仍需要查看、编辑和对比代码内容。因此，系统必须具备基础而完整的代码工作台能力。具体而言，平台应支持文件树浏览、文件内容编辑、当前文件定位、修改状态标识、文件搜索、差异查看和历史变更跟踪等功能。

这一需求的核心不在于做一个完整 IDE，而在于建立对话生成与代码可见性之间的桥梁。用户需要知道模型修改了哪些文件、修改前后差异如何、当前工作文件是什么、是否存在尚未保存的变更，以及这些变更是否已被运行时采用。只有这样，平台才能避免“看不见的自动化”导致的不确定感。

### 3.2.6 历史记录与数据持久化需求

由于开发辅助任务通常具有连续性，系统应能够保存聊天记录、项目快照和部分运行上下文，支持用户离开页面后再次恢复会话。否则，用户每次进入系统都需要重新描述需求和恢复上下文，将显著降低使用价值。

结合项目现有实现，该需求主要包括：聊天消息历史保存、聊天描述与元数据保存、项目快照保存、聊天复制与恢复、URL 与会话标识映射，以及本地文件状态与远程账户状态的协同管理。对于登录用户，系统还应支持基本的账户识别与配置保存；对于未登录用户，也应至少保证本地会话在合理范围内可恢复。

### 3.2.7 多模型配置与切换需求

本系统的一个显著特点是支持多个 AI 提供商统一接入，因此模型配置与切换应被视为独立功能需求，而不是附属设置项。系统应允许用户查看可用模型列表、选择当前模型与提供商、配置相应 API Key 或本地模型地址，并在不同模型之间进行切换。

同时，由于不同模型能力和接口存在差异，系统还应支持模型列表动态更新、启用状态配置、错误检查和可用性提示。对于研究型项目而言，这一需求不仅服务用户体验，也关系到系统的可扩展性和后续实验比较能力。

### 3.2.8 第三方服务接入与部署扩展需求

为了增强系统的完整性与实用性，平台还应支持部分第三方服务接入和部署扩展能力。从项目已有实现看，系统已具备 GitHub、GitLab、Vercel、Netlify、Supabase、MCP 等接入基础，并支持导入、导出、部署和账户配置等操作。

因此，系统在此方面至少应满足以下需求：支持从外部代码仓库导入项目；支持将当前生成结果导出或部署到第三方平台；支持配置 MCP、云服务和本地模型等扩展能力；支持在设置面板中对这些外部能力进行集中管理。需要指出的是，这部分功能虽然不是平台最基础的闭环能力，但对于体现系统工程完整度和平台扩展价值具有重要意义。

### 3.2.9 桌面端运行需求

除浏览器端运行外，系统还应具备向桌面端迁移的能力，以适应不同使用场景和部署方式。该需求并不要求桌面端与 Web 端完全独立实现两套系统，而是要求现有技术栈具备可封装性、可迁移性和本地能力接入空间。

基于项目现状，桌面端需求主要体现为：能够通过 Electron 加载现有应用；能够在主进程中管理窗口与本地运行环境；能够保留核心聊天、编辑、预览和设置功能；能够扩展本地菜单、自动更新和会话管理等桌面能力。

## 3.3 系统非功能需求分析

### 3.3.1 易用性需求

本系统面向的用户既包括具有一定开发经验的使用者，也包括希望通过自然语言快速生成原型的初学者。因此，平台在交互层面应保持低门槛。具体而言，系统界面应尽量将高频功能集中在用户可见区域，例如消息输入、模型选择、工作台切换和预览反馈；配置复杂度较高的部分应收纳到设置面板中统一管理，避免打断主流程。系统还应通过状态提示、进度展示和错误信息说明，降低用户对系统内部执行过程的不确定感。

### 3.3.2 可维护性需求

由于系统涉及前端界面、模型接入、运行时管理、持久化服务和桌面端扩展等多个层次，若代码组织不清晰，将很难开展后续维护和功能扩展。因此，系统在工程层面应具备良好的模块划分、明确的职责边界和可追踪的数据流。项目当前已按照组件、路由、运行时、服务、状态存储等维度进行分层，这说明系统实现已经具备一定可维护性基础。

### 3.3.3 可扩展性需求

智能辅助平台具有明显的演进性。随着模型供应商增多、部署平台变化和工具链扩展，系统必须能够在不大幅重构核心架构的前提下新增能力。因此，平台需要在模型提供者抽象、第三方服务接入、设置项扩展和工作台功能插拔等方面预留接口。例如，多模型注册机制、第三方平台路由接口和模块化设置面板都体现了这类需求。

### 3.3.4 安全性需求

由于系统需要处理 API Key、用户配置、代码内容和运行命令，因此安全性不可忽视。首先，平台应尽量避免将敏感信息以不受控形式暴露在前端界面中。其次，代码运行环境应具备隔离性，避免生成代码直接影响宿主系统。再次，系统应对外部服务调用结果、运行异常和配置错误进行边界控制，降低非预期行为带来的风险。浏览器端沙箱运行、受控设置面板以及服务端路由边界，都是满足此类需求的重要手段。

### 3.3.5 兼容性需求

本系统需要在现代浏览器环境中稳定运行，并在桌面端具备可迁移性。因此，兼容性需求包括浏览器兼容、运行环境兼容和平台兼容三个方面。浏览器端需要保证主要功能在主流浏览器中可用，尤其是聊天交互、编辑器和预览模块；运行环境方面需要兼顾开发模式、构建模式和桌面封装模式；平台方面则需要支持 Windows、macOS 和 Linux 等常见环境。

### 3.3.6 响应性能需求

系统的用户体验高度依赖响应速度。若自然语言提交后长期无反馈，或生成完成后预览迟迟无法启动，将直接影响平台可用性。因此，系统在模型响应、文件处理、终端输出、视图切换和预览加载等方面都应具备较好的响应能力。即便在某些阶段无法立刻返回最终结果，也应提供流式内容、阶段性进度或异常提示，以维持交互连续性。

## 3.4 业务流程分析

### 3.4.1 用户角色分析

从系统使用方式出发，本文平台中的核心角色可以概括为两类。第一类是平台用户，包括匿名使用者和登录用户。匿名使用者主要关注即时体验与本地会话，登录用户则还需要账户识别、配置保存和外部服务绑定等能力。第二类是外部平台服务，包括大语言模型服务提供商、部署平台、代码托管平台和 Supabase 等后端服务。这些对象虽然不是“人”的角色，但在业务流程中承担关键参与方职责。

在论文需求分析中，不必将所有外部服务等同于独立业务角色，而应将其视为系统边界外的重要协作对象。系统设计需要考虑这些对象与核心平台之间的交互方式、输入输出结构以及失败处理策略。

### 3.4.2 用户使用流程分析

用户使用系统的一般流程如下。首先，用户进入平台，在必要时完成模型配置或登录操作。其次，用户通过输入框描述需求，也可以附加图片、URL 内容或模板上下文。随后，系统将请求发送至聊天接口，由模型完成响应并产出文件操作、命令操作或文本解释。接着，工作台根据解析结果执行相应动作，将生成文件写入项目、在浏览器端安装依赖并启动运行服务。之后，用户可在文件树、编辑器、终端和预览面板中查看结果，并根据预览反馈继续发起新一轮修改请求。最后，用户可选择导出、部署、保存会话或关闭页面，系统则负责保存必要的历史记录与快照。

这一流程体现出本系统与普通聊天型 AI 工具的区别，即用户并不是在“提问并获得答案”，而是在“描述需求并推动一个项目持续演化”。

### 3.4.3 自然语言到应用生成的业务流程

从系统内部角度看，平台的核心业务流程可以概括为以下顺序：用户输入自然语言需求后，前端收集当前聊天消息、已存在文件、模型配置和辅助上下文；随后，服务端接口根据消息与文件上下文组织模型请求，并在需要时进行上下文优化和工具调用处理；模型返回文本与结构化产物后，消息解析器将文件操作、命令操作和产物卡片提取出来并交由工作台管理；工作台通过动作执行器将文件写入浏览器端文件系统，并调用运行时安装依赖和启动服务；最终，系统在预览区域展示运行结果，并把执行状态同步回对话区和工作台。

该流程表明，系统的业务主线贯穿了自然语言理解、模型推理、结构化动作解析、浏览器端运行和可视化反馈五个阶段，每个阶段都必须设计明确的输入输出边界，才能保证整体闭环稳定。

### 3.4.4 代码运行与反馈迭代流程

当生成内容被写入项目后，系统还要经历一个运行与验证循环。具体过程为：运行时安装项目依赖，执行启动命令并侦测可用预览端口；预览面板加载运行结果，并将错误信息、控制台异常或预览异常回传到平台；用户观察运行结果后，决定继续通过自然语言描述修改需求，或在编辑器中直接调整代码；系统再以更新后的上下文进入下一轮生成与运行。

这一路径说明，本平台的本质是“生成—验证—修正”的迭代系统，而不是一次性静态生成器。因此，需求分析必须把运行反馈和持续迭代视为核心业务，而非附加特性。

## 3.5 可行性分析

### 3.5.1 技术可行性

从技术角度看，构建本系统具备较强可行性。首先，Remix、React 与 Vite 已形成成熟的现代 Web 全栈开发组合，能够支撑页面路由、服务端逻辑和高效前端工程化。其次，大语言模型和统一模型接入框架已经能够稳定支持多供应商调用，为自然语言解析和代码生成提供基础。再次，WebContainer 提供了浏览器端运行 Node.js 项目所需的关键能力，使“生成后立即运行”的目标不再停留在概念阶段。最后，Supabase 和 Electron 分别提供了持久化与桌面化扩展基础，降低了系统其余基础设施的实现成本。

结合项目当前已有实现情况可以看出，上述关键技术并非难以落地的前沿概念，而是已经在同一工程中实现了协同使用，因此本文课题在技术上是可实施的。

### 3.5.2 经济可行性

从经济角度看，本系统主要采用开源框架与可按需接入的第三方服务，整体研发成本较为可控。前端框架、工程工具、组件库和 Electron 等核心基础能力均可通过开源方案获得；浏览器端运行时减少了持续维护远程容器资源的压力；AI 调用成本则主要取决于用户选择的模型服务和调用频率。对于论文型项目和原型系统而言，这种成本结构是可接受的。

此外，系统支持多模型与多平台接入，也意味着后续可以根据使用场景灵活调整外部服务依赖，避免过度绑定单一商业平台，从而在一定程度上提升整体经济可行性。

### 3.5.3 实施可行性

从实施角度看，本系统具备较清晰的分模块推进路径。系统可先实现聊天交互、模型调用和工作台基础功能，再逐步接入浏览器端运行时、持久化服务和部署扩展能力。这种递进式实施方式有利于在项目开发过程中逐步验证关键假设，降低一次性集成过多功能带来的风险。

同时，项目当前已经形成较完整的代码基础，包括对话界面、工作台、模型管理、WebContainer 封装、部署接口和 Electron 主进程结构等，这说明后续论文撰写不仅有理论依据，也有实际系统作为支撑，实施可行性较高。

## 3.6 本章小结

本章从系统建设目标、功能需求、非功能需求、业务流程和可行性五个方面对平台进行了分析。需求分析表明，本文所研究的平台并非单一聊天应用，而是一个围绕“自然语言输入—代码生成—浏览器内运行—结果反馈—持续迭代”构建的智能开发辅助系统。其核心需求集中在自然语言交互、项目级代码生成、浏览器端运行、工作台联动、历史持久化、多模型接入以及部署扩展等方面；其非功能需求则强调易用性、可维护性、可扩展性、安全性、兼容性和响应性能。

上述需求分析为后续系统设计提供了明确依据。下一章将在此基础上进一步给出系统总体架构、模块划分、数据库设计与接口设计，从而将需求转化为可实现的系统方案。
