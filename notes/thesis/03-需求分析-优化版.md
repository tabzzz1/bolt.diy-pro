# 第三章 需求分析

## 3.1 系统建设目标

本文研究的系统并不是一个单纯的聊天工具，也不是传统意义上的在线代码编辑器，而是一个面向软件原型快速构建的自然语言驱动智能辅助平台。其目标并不止于“让模型生成一段代码”，而在于打通从需求描述、代码生成、运行验证到结果反馈的完整链路，使用户能够在同一工作台内完成从想法表达到账户配置、从代码修改到界面预览的连续操作。

从平台定位看，本文平台主要服务于三类需求场景。其一是教学与入门场景，即帮助初学者绕开繁琐的环境搭建和依赖配置过程，将注意力更多放在需求理解、界面结构和实现思路上。其二是原型验证场景，即缩短从创意描述到可运行演示页面之间的时间路径，提高产品构思和功能试验的效率。其三是研究与工程实践场景，即验证“自然语言驱动 + 浏览器端运行时 + 一体化工作台”这一技术路线在真实系统中的可行性。

围绕这一总体定位，系统建设目标可以概括为以下几个方面：一是具备稳定的自然语言交互能力，能够理解用户以文本方式提出的开发需求；二是具备项目级代码生成能力，生成结果不仅包含页面片段，还应覆盖文件结构、依赖关系和运行命令；三是具备浏览器内运行和结果预览能力，使生成代码可以被立即验证；四是具备持续迭代能力，允许用户结合对话结果和预览反馈继续修正项目；五是具备一定的持久化与扩展能力，支持历史记录保存、模型切换、账户接入和桌面端迁移。

需要说明的是，这里的建设目标并不要求系统在每一次生成中都达到完全自动化和一次成功，而是强调平台应提供一个可追踪、可验证、可继续修正的工程闭环。这一点决定了后续需求分析不能只停留在“能否生成代码”，而必须进一步关注运行时、工作台、状态管理和错误反馈等支撑能力。

## 3.2 系统功能需求分析

### 3.2.1 核心业务需求

系统最核心的功能需求集中在“输入、生成、运行、反馈”这条主线上。

首先，平台应支持自然语言输入与多轮对话。用户在真实开发过程中往往不会一次性给出完整需求，而是边观察结果边补充限制条件、修正样式要求或提出局部修改意见。因此，系统需要支持连续会话，并能够在后续生成中参考前文消息和当前项目状态。除了基础文本输入外，聊天区域还应能够承载模型切换、错误提示、结构化产物展示以及附加上下文输入等操作。

其次，平台应支持项目级代码生成。与一般问答系统不同，本文平台的目标是形成可运行的工程产物，因此输出内容不能只是若干零散代码片段，而应包含文件写入动作、目录结构、依赖声明和必要的启动命令。系统还应能够基于现有文件进行增量修改，而不是在每一轮对话中重复生成相互冲突的完整项目。

再次，平台应支持浏览器内编译、安装和运行。若生成结果只能停留在文本层面，用户仍需要手动复制代码、安装依赖并自行排查运行问题，这将削弱系统的实际价值。因此，系统需要在浏览器侧完成项目文件写入、依赖安装、命令执行和开发服务启动，并将执行状态回显到工作台中。

最后，平台应提供结果反馈与继续迭代能力。代码生成之后，用户需要立即看到运行效果，判断当前页面结构、交互逻辑或样式实现是否符合预期；若存在问题，系统还应支持用户基于运行结果继续发起修改请求。这意味着预览区域、终端输出、对话区和编辑区之间不能相互割裂，而应形成可联动的整体。

从这一层面看，本文平台的核心业务需求并不是若干孤立功能的简单叠加，而是围绕开发闭环构成的连续流程。只要其中某一环节失效，例如生成内容无法写入项目、项目无法启动、错误无法回传，整个系统的使用体验就会显著下降。

### 3.2.2 工作台与辅助功能需求

在核心闭环之外，系统还需要一组辅助功能来保证开发过程的可见性和连续性。

首先是代码编辑与文件管理能力。尽管平台强调自然语言驱动，但在实际使用中，用户仍需要查看当前项目有哪些文件、模型修改了哪些内容、某个文件的前后差异是什么，以及运行结果究竟对应哪一部分代码。因此，系统至少应支持文件树浏览、文件切换、代码编辑、差异查看和运行中关键文件的定位。这里的目标不是构建一个完整 IDE，而是让用户对模型行为和项目状态保持足够清晰的感知。

其次是历史记录与状态持久化能力。开发辅助任务通常具有连续性，用户往往需要在一段时间后回到原有项目继续修改。如果系统不能保存聊天历史、项目快照和必要配置，那么每次重新进入平台都要重新说明背景，使用成本会明显上升。因此，系统应支持会话记录保存、项目状态恢复以及局部配置持久化。对于登录用户，还应支持更稳定的账户识别和配置同步。

再次是多模型管理能力。考虑到不同模型在响应速度、成本、代码生成能力和稳定性方面存在差异，系统应允许用户查看可用模型、切换当前提供商、配置对应密钥或服务地址，并在接口层面对不同模型做统一封装。对于论文项目而言，这一能力不仅提升系统可用性，也有助于体现平台在架构上的扩展性。

最后是第三方接入与部署扩展能力。严格来说，这部分并不是平台形成开发闭环的前提条件，但它能够明显提升系统的完整性。比如，将项目导出到外部仓库、部署到云平台、接入外部服务或在桌面端运行，都能体现该平台不是一次性实验页面，而是具备进一步落地和扩展的基础。由于这些功能在当前系统中属于增强项，因此在需求分析中不必给予与核心链路相同的篇幅，但仍应保留其必要性说明。

## 3.3 系统非功能需求分析

### 3.3.1 易用性与响应效率

本文平台的目标用户既包括具备一定开发基础的使用者，也包括希望借助自然语言快速构建原型的初学者。因此，系统在易用性上应尽量减少不必要的学习成本。具体而言，输入区域、模型选择、工作台切换、预览反馈等高频能力应保持清晰可见，而配置复杂度较高的内容则更适合统一收纳到设置区域。对于用户来说，最重要的不是理解系统内部实现细节，而是明确知道“现在系统在做什么”“哪里出了问题”“下一步能做什么”。

与易用性紧密相关的是响应效率。系统并不一定要求每一步都在极短时间内完成，但必须避免长时间无反馈。特别是在模型生成、依赖安装和服务启动阶段，如果界面没有流式内容、阶段提示或错误说明，用户会很容易产生中断感。因此，平台应尽可能提供流式响应、状态回显和结果预览，维持交互连续性。

### 3.3.2 可维护性与可扩展性

由于系统同时涉及前端界面、服务路由、模型调度、浏览器运行时、状态管理、持久化服务和桌面端封装，工程结构若缺乏清晰分层，后续开发将很难持续推进。因此，系统需要具备良好的模块边界，使聊天交互、模型调用、运行时管理、文件存储和平台扩展等部分能够相对独立地演进。

此外，智能开发平台本身具有持续变化的特点。模型供应商会变化，部署平台会变化，工作台能力也会不断扩展，因此系统应预留新增模型、新增外部服务和新增配置项的空间。换言之，本文平台不能只满足当前一次性开发需求，还应具备继续演进的工程基础。

### 3.3.3 安全性与兼容性

安全性主要体现在两个方面。一方面，系统会处理 API Key、用户配置、代码内容和执行命令等敏感信息，因此需要避免无控制地暴露关键配置；另一方面，系统运行的是模型生成代码，运行环境必须尽可能与宿主系统隔离，降低异常代码对本机环境的直接影响。浏览器端沙箱运行机制正是支撑这一需求的重要基础。

兼容性则主要体现在浏览器环境、构建环境和平台环境三个层面。系统应能够在主流现代浏览器中完成基本交互和预览流程，同时保证开发模式、生产构建模式以及桌面端封装模式之间具有较好的迁移性。对于毕业设计项目而言，兼容性并不要求覆盖所有极端环境，但至少应保证主要使用场景下的稳定可用。

## 3.4 业务流程分析

### 3.4.1 平台总体使用流程

从用户视角看，系统的使用流程相对清晰。用户进入平台后，首先完成必要的模型配置或登录操作，然后在输入区以自然语言描述功能需求，也可以补充图片、URL 内容或模板信息。系统接收请求后调用相应模型，并将返回结果解析为文本说明、文件操作和命令操作。随后，工作台根据这些结构化结果更新项目文件，在浏览器内安装依赖、执行命令并尝试启动服务。服务启动后，用户即可在文件树、编辑器、终端和预览区域中查看结果，并根据运行反馈继续发起下一轮修改请求。

这一流程表明，平台的使用逻辑并不是传统的“提问—回答”模式，而更接近“提出需求—生成项目—观察结果—继续修正”的开发活动流程。因此，在论文中将它归类为智能开发辅助平台，比归类为普通聊天系统更为准确。

### 3.4.2 核心闭环流程

从系统内部角度看，平台的核心业务由五个阶段构成：需求输入、上下文组织、模型生成、动作执行和结果反馈。

其中，需求输入阶段负责收集用户消息和当前会话状态；上下文组织阶段负责组合历史消息、已有文件和模型配置，为后续生成提供尽可能合理的上下文；模型生成阶段输出文本解释以及结构化的文件和命令信息；动作执行阶段则将这些结果真正落到项目文件和运行环境中；结果反馈阶段把运行状态、终端信息和预览结果返回给用户，从而进入下一轮迭代。

这一流程之所以重要，是因为它解释了本文平台与传统代码补全工具的差别。后者往往只覆盖“生成建议”这一局部环节，而本文平台试图把“生成之后怎么办”也纳入平台能力之中。正因为如此，需求分析必须把运行时、工作台和错误反馈视为主流程的一部分，而不能简单归入附属功能。

## 3.5 可行性分析

### 3.5.1 技术可行性

从技术实现角度看，本文平台具备较强可行性。Remix、React 与 Vite 可以提供成熟的全栈 Web 开发基础；大语言模型与统一模型接入机制能够支撑自然语言理解和代码生成；WebContainer 使浏览器内运行 Node.js 项目成为现实；Supabase 和 Electron 则分别为用户认证、持久化管理和桌面端扩展提供了现成支撑。这些技术并非停留在概念层面，而是已经能够在同一项目中协同使用，因此技术路线是成立的。

### 3.5.2 经济可行性

从经济角度看，本文平台主要建立在开源框架和按需接入的第三方服务之上，整体研发成本相对可控。对于毕业设计和原型平台而言，这样的技术选型既能保证实现深度，也不会引入过高的基础设施投入。模型调用成本虽然会随着使用频率增加而上升，但由于系统本身支持多模型接入，因此后续可根据场景灵活调整成本结构。

### 3.5.3 实施可行性

从实施路径看，系统可以按模块逐步推进，先完成聊天交互、模型接入和工作台基础能力，再补充浏览器端运行时、数据持久化和部署扩展功能。当前项目已经形成较为完整的代码基础，这意味着论文分析并不是建立在抽象设想之上，而是有实际系统作为支撑，实施可行性较高。

## 3.6 本章小结

本章围绕系统建设目标、功能需求、非功能需求、业务流程和可行性对平台进行了分析。与普通聊天型 AI 工具相比，本文研究的对象更强调开发闭环，即在同一平台内完成需求描述、代码生成、运行验证和持续迭代。基于这一定位，系统的核心需求应放在自然语言交互、项目级代码生成、浏览器端运行和结果反馈联动上，而文件管理、历史持久化、多模型切换以及第三方接入则构成支撑平台完整性的辅助能力。

从整体上看，这样的需求结构更符合本文平台的真实特点，也为下一章的系统设计提供了更明确的落点。后续系统设计将围绕总体架构、功能模块、数据存储和接口交互等内容展开，把本章提出的需求进一步转化为可实现的系统方案。
