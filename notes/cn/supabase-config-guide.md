# Supabase 配置梳理：服务端 vs 用户自行配置

本文档用于理清项目中两套独立 Supabase 配置的功能边界、存储位置及其本质区别。

---

## 1. 配置体系概览

项目中存在两套完全物理隔离的 Supabase 配置，分别服务于“应用基础架构”和“用户资源管理”。

| 特性 | **服务端配置 (Infrastructure)** | **用户自行配置 (Management)** |
| :--- | :--- | :--- |
| **面向对象** | 开发者（您）名下的项目 | 终端用户（或客户）名下的项目 |
| **存储位置** | 云端环境变量 / `.env` 文件 | 浏览器 `LocalStorage` (客户端持久化) |
| **认证方式** | `Anon Key` / `Service Role Key` | `Personal Access Token` (PAT) |
| **主要功能** | 支撑应用运行（登录、个人资料、日志） | 管理外部资源（监控负载、查账单、跳控制台） |
| **数据归属** | 您拥有最高管理权限 | 仅用户自己可见，您无法获取 |

---

## 2. 服务端配置 (系统级 / 后台级)

这是应用运行的“地基”，由开发者在部署时配置。

*   **配置项：** `VITE_SUPABASE_URL` 和 `VITE_SUPABASE_ANON_KEY`。
*   **对应代码：** [app/lib/services/supabaseAuth.ts](../app/lib/services/supabaseAuth.ts)
*   **核心功能：**
    *   **用户认证 (Auth)：** 处理用户的注册与登录逻辑。
    *   **用户画像 (Profiles)：** 存储用户的头像、昵称、应用偏好设置。
    *   **对象存储 (Storage)：** 承载用户上传的资源或应用生成的静态内容。
    *   **系统日志：** 记录应用层面的运行审计日志。
*   **总结：** 这里的数据库是**您的**。所有用户产生的业务数据（如账号信息）都汇聚于此。

---

## 3. 用户自行配置 (个人级 / 管理级)

这是用户在聊天界面右下角或设置面板中输入的 **Supabase 管理令牌**。

*   **配置项：** 用户个人的 `Management API Token`。
*   **对应代码：** [app/lib/stores/supabase.ts](../app/lib/stores/supabase.ts) 和相关 API 路由。
*   **核心功能：**
    *   **项目列表管理：** 允许用户在您的应用内直接看到他自己在 Supabase 官网上创建的所有项目。
    *   **状态监控：** 实时展示用户项目的健康状态、数据库大小、API 调用频率等。
    *   **便捷运维：** 提供一键跳转到用户自己项目控制台（SQL 编辑器、用户列表、函数定义）的快捷入口。
*   **总结：** 这是一个**管理集成工具**。它不用于统计“用户在您的应用里聊了什么”，而是方便开发者（您的应用用户）在开发过程中快速监控和调整他们**自己的**基础设施。

---

## 4. 常见疑问解答 (FAQ)

### Q：用户在右下角配置 Supabase 是为了统计聊天数据吗？
**A：不是。** 它的目的是为了让用户（通常是开发者）能够在一个界面内连接并管理他们自己的云资源。它统计的是“基础设施运行状态”，而不是“业务聊天记录”。

### Q：作为开发者，我能看到用户自行配置的数据吗？
**A：不能。** 这些 Token 存储在用户的浏览器 `LocalStorage` 中，发送请求时通过 API 代理透传给 Supabase 官网。这些数据是私密的，除非用户显式将其作为变量注入到生成的代码中，否则您无法在服务端获取。

### Q：两套配置会产生数据交叉吗？
**A：不会。** 它们是并行运行的：
*   用户登录您的应用用的是您的 Supabase 项目。
*   用户在界面里查看项目列表显示的是他自己的 Supabase 项目。
两者物理隔离，互不干涉。
